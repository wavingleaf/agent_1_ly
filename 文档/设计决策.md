## 决策 1：为什么用 LangChain + LangGraph（代码优先）？

**可选项**：
- A) LangChain + LangGraph（代码框架）
- B) Dify / Langflow / Flowise（低代码拖拽）
- C) 纯 OpenAI SDK + 自造轮子
- D) CrewAI / AutoGen（多 Agent 框架）

**质疑**（grill 模式）：
> "LangChain 一年变三个大版本，学它值得吗？低代码平台拖几下就能跑，为什么写代码？"

**回应**：

| 维度 | 低代码平台 | LangChain+LangGraph |
|------|-----------|-------------------|
| 学习深度 | 拖拽完不知道内部在干什么 | 每个节点、每条边都是自己写的 |
| 定制上限 | 平台提供什么就只能用什么 | 无限——手写 StateGraph 可加任意节点 |
| 版本风险 | 平台停更/收费/Docker 炸了一样黑盒 | 开源框架，即使 API 变也有迁移路径 |
| 本项目的目标 | ❌ 快速出产品 | ✅ **理解 Agent 底层原理** |

**结论**：LangChain 1.0（2025年发布）提供了两层抽象——高层 `create_agent()` 和中层手写 StateGraph。本项目当前使用手写 StateGraph（足够灵活以支持远期控制台指令辅助功能的复杂编排）。

## 决策 2：为什么 Python ≥ 3.11？

**现象**：系统默认 `python` 指向 Python 3.7（Anaconda），项目用 `D:/Apps/Python/python.exe`（3.12.6）。

**原因**：
- LangGraph 的 `astream()` 依赖 `asyncio` + `ContextVar` 传播
- Python 3.10 以下 `ContextVar` 在 `asyncio.to_thread()` 等跨线程 async 场景有 bug
- LangChain 1.0 没有官方声明最低版本，但社区和生产部署指南统一要求 3.11+

**教训**：环境问题最容易浪费生命——PATH 里的 Anaconda 3.7 排在系统 Python 3.12 前面，每个命令都要写全路径。以后同类项目应优先配 `.venv` 并用 `activate` 隔离。

## 决策 3：为什么同时提供三种 graph 构建方式？

`graph.py` 中有三个函数：

```python
graph                  # build_graph() → 手写 StateGraph + MemorySaver
graph_persistent       # build_graph_with_persistence() → 手写 + SqliteSaver
agent_with_middleware  # build_agent_with_middleware() → create_agent() + 中间件
```

**质疑**：
> "一个项目三种 Agent，不混乱吗？生产环境到底用哪个？"

**回应**：

这是一个**教学项目**，不是生产系统。「三个阶段」对应学习者的认知阶梯：

| 实例 | 学习者看到什么 | 理解什么 |
|------|-------------|---------|
| `agent_with_middleware` | `create_agent(model, tools, system_prompt)` 一行 | "Agent 就这么简单？" |
| `graph` | `StateGraph` + `add_node("agent")` + `add_conditional_edges(...)` | "哦，底层是状态图！" |
| `graph_persistent` | 同上 + `SqliteSaver(conn)` | "Checkpointer 可以换的！" |

**如果是生产系统**：只需要 `graph_persistent` 一个（或者等价地，删掉另外两个）。教学项目的"冗余"是有意的——每个版本是一级台阶。

## 决策 4 & 5：LLM 节点命名差异

`create_agent()` 编译后的 graph 中，LLM 调用节点叫 `model`，而手写版叫 `agent`。

**起因**：测试 `test_graph_builds()` 中检查 `"agent" in g.nodes` 时发现在 `agent_with_middleware` 上失败。

**为什么 create_agent() 叫 model？**
- `create_agent()` 内部用 LangGraph 的 `create_react_agent` 遗留命名
- 节点名反映了 LangChain 团队的架构视角：这个节点"调用模型"，不是"Agent 的全部"
- 中间件的 `before_model` / `after_model` 也是围绕"模型调用"这个操作命名的

**为什么手写版故意叫 agent？**
- 教学场景下 `agent` 比 `model` 直观："Agent 节点在做推理"
- 等学到中间件那一步，自然会发现 `before_model` 这个钩子名和节点名 `model` 是对齐的，形成概念闭环

**教训**：测试中不应假设 `create_agent()` 的节点名和手写版一致。已在 `test_graph_builds` 中分开检查。

## 决策 6：SSE vs WebSocket

**质疑**：
> "WebSocket 是全双工，SSE 是半双工。Agent 对话中用户可能想中途发新消息打断当前回答，这不就需要全双工吗？"

**回应**：

| 维度 | SSE | WebSocket | 对本项目的影响 |
|------|-----|-----------|---------------|
| 协议复杂度 | HTTP，浏览器原生 `EventSource` | 需升级协议，需专门库 | SSE 实现量少 ~70% |
| 代理兼容性 | 所有 HTTP 代理天然支持 | 需代理显式支持 WebSocket | **生产环境的关键差异** |
| 自动重连 | `EventSource` 内置 | 需手动实现 | SSE 开箱即用 |
| 客户端→服务端 | 需要另发 HTTP 请求 | 同一连接双向 | 本项目够用 |
| 打断能力 | 前端 `AbortController` + 新请求 `new thread_id` | 发送 stop 帧 | 两种方式都能实现 |

**打断当前回答的方案**（SSE 下）：
1. 前端 `fetch()` 绑 `AbortController` → `.abort()` 关闭 HTTP 连接
2. 后端 `GeneratorExit` 被触发 → 资源回收
3. 如果用户此时发新消息 → 同一个 `thread_id`，LangGraph 从最后一个 checkpoint 继续

结论：SSE 足够。等需要"用户在同一个回答流中交互式引导 Agent"时再切 WebSocket——那是另一个问题。

## 决策 7：SqliteSaver（同步版）→ AsyncSqliteSaver（已更新）

**原决策**（2026-07-03）：选同步版 SqliteSaver，因为模块级全局实例不能是 async context manager。

**更新**（2026-07-05）：绕过 `from_conn_string()` 的 context manager 限制，直接传 `aiosqlite.Connection` 给 `AsyncSqliteSaver` 构造函数。在 FastAPI `lifespan` 中异步初始化，通过 `init_graph()` 替换全局 graph 实例。`/chat` 端点由 `graph.invoke()` 改为 `await graph.ainvoke()`（同步版会 500 报错）。

**关键发现**：
- `AsyncSqliteSaver.from_conn_string()` 是 context manager，不能用于模块级 init
- 直接传 `aiosqlite.Connection` 给构造函数可绕过，lifespan 中 await 一次即可
- 同步 `invoke()` 与 AsyncSqliteSaver 不兼容，必须改用 `ainvoke()`

## 决策 9：中间件钩子的选择

LangChain 1.0 提供 8 个中间件钩子，本项目只用了 3 个：

```
before_model      ← LLM 调用前记录开始时间
after_model       ← LLM 调用后计算耗时 + token
wrap_tool_call    ← 工具调用前后各记一行日志
```

**没用的钩子**（及原因）：

| 钩子 | 为什么不实现 |
|------|------------|
| `before_agent` | 本项目无 agent 级初始化需求（加载记忆等） |
| `after_agent` | 无 agent 级清理需求 |
| `wrap_model_call` | 日志中间件不需要拦截/修改请求 |
| `awrap_model_call` / `awrap_tool_call` | 异步版本，同步版已覆盖需求 |

**选择原则**：覆盖"可观测性"的最小钩子集。每个钩子都在 notebook 03 中有验证——日志输出清晰可见。

## 决策 10：四子包架构

```
src/
├── agent/        ← 编排核心（state + graph）
├── config/       ← 配置加载（pydantic-settings）
├── tools/        ← 工具定义（@tool）
└── memory/       ← 持久化（checkpointer factory）
```

**质疑**：
> "四个子包加起来不到 10 个文件。参考项目 SuperMew 有 9 个子目录、60+ 文件。你是不是过度简化了？"

**回应**：结构跟随规模。参考项目是多用户生产系统，需要 `api/`、`schemas/`、`infra/`、`indexing/` 等层。本项目是**单用户学习项目**，10 个文件已经表达了完整概念：

| 参考项目子目录 | 本项目对应 | 备注 |
|---------------|----------|------|
| `chat/` (5 文件) | `agent/` (2 文件) | 无多用户会话管理 |
| `rag/` (2 文件) | `tools/` (1 文件) | 无复杂检索管线 |
| `indexing/` (6 文件) | 无 | 不对接文档 |
| `api/` (5 文件) | `main.py` (顶层) | 单文件 FastAPI |
| `infra/` (3 文件) | `memory/` (1 文件) | 仅 SQLite checkpointer |
| `schemas/` (3 文件) | 无 | pydantic-settings 已够 |

**升级条件**（如 RAG 项目的 `src架构设计.md` 提出的）：当某个文件膨胀到 ~200 行或出现 3+ 个内部辅助函数时，升级为子包。当前所有文件均在 100 行以下。

## 决策 11：grep 精确检索而非 RAG 向量检索

**决策日期**：项目立项时就隐含此方向，2026-07-06 正式写入 ADR。

**为什么 grep 更适合这个场景：**

| 维度 | RAG 向量检索 | grep 精确匹配 |
|------|------------|-------------|
| 查函数名/方法名 | 可能漏（语义近似 ≠ 字符串匹配） | **精准命中** `function X:SetDomestication` |
| 查组件字段 | 向量不区分 `inst.` 和 `self.` | **行号+上下文**一目了然 |
| 结果可靠性 | 概率性，可能遗漏关键定义 | **确定性**，只要字符串匹配就能找到 |
| 依赖条件 | 需预建索引、embedding 模型、ChromaDB | 仅需文件系统 |

### LLM 提供「候选」，grep 提供「验证」

从"驯化度+7"推理出搜索 `domesticatable|SetDomestication|GetDomestication`，LLM **不是在查文档**——它是**基于训练语料做统计推断**。模型在训练时见过 DST 的 Lua 源码、Wiki 讨论、Mod 社区帖子，所以"驯化度"和 `domesticatable` 在它的参数空间中形成了高概率关联。

**但这个推断是不可靠的。** 具体来说有三种风险：

| 风险 | 示例 | 后果 |
|------|------|------|
| **语料过时** | 游戏版本迭代后 `SetDomestication` 改名为 `SetDomesticationPercent`，但训练数据中是旧版 | grep 搜不到目标符号 |
| **语料不全** | 某个冷门组件（如 `terraformer`）根本没出现在训练语料中 | LLM 完全不知道要搜什么 |
| **幻觉** | LLM 把另一个 Lua 项目的 API 安到了 DST 头上 | grep 搜到的是无关代码或空结果 |

### 工作流：循环验证而非单次推理

```
用户需求
  → LLM 推理候选搜索词（基于训练语料 + 中英文术语常识）
  → grep 验证：候选符号是否真的存在于本地源码中？
  → 命中 → read_file 阅读上下文 → 确认方法签名 → 拼装命令
  → 未命中 → LLM 换一组搜索策略重试
       （例如：搜更宽泛的关键词、搜组件文件名、
        搜可能相关的 TUNING 常量）
```

LLM 的角色是**提出假设**，不是**给出答案**。grep 才是最终裁决——本地源码是唯一的地面真相。

**工具链需求**：Agent 必须能够**多轮 grep**（第一轮命中率不高时应自动调整搜索词重试），而不是一次 grep 失败就放弃。

## 决策 12：类 grep 垂直工具合并为一个 `dst_data_lookup`

**决策日期**：2026-07-06（grill-with-docs 讨论）。

### 背景

`控制台指令辅助编写_TODO.md` 原计划新增 3 个独立工具：
- `tuning_lookup` — 查 TUNING 常量定义值
- `prefab_list` — 验证 prefab ID 合法性
- `chinese_name_lookup` — 中文名 → prefab ID

三者都是"在已知数据文件中做模式匹配"，本质上都是 grep 的垂直定制变体。但实测数据显示 DeepSeek 的工具选择准确率随工具数量增加而下降：

```
工具数:  1    2    3    4    5    6    7    8
准确率: 94%  90%  83%  78%  72%  68%  65%  63%
```

（数据来源：DeepSeek V3 实测。V4-Pro 没有独立的工具路由基准，但下降趋势是概率空间的固有属性。）

### 策略

将 3 个工具合并为 1 个 `dst_data_lookup`，用 `category` 参数做子功能分发：

```python
@tool
def dst_data_lookup(pattern: str, category: str = "auto") -> str:
```

| category | 数据源 | 用途 |
|----------|--------|------|
| `"tuning"` | `tuning.lua` | 查常量定义值 |
| `"prefab"` | `prefablist.lua` | 验证 prefab ID 是否存在 |
| `"chinese"` | `chinese_s.po` | 中文名 → prefab ID |
| `"auto"` | 自动判断 | 根据 pattern 特征自动路由 |

### `chinese_s.po` 解析的关键发现

`.po` 文件中的 `msgctxt` 行本身就编码了条目类型——不需要正则 lookbehind 或交叉验证 `prefablist.lua`：

```
# 物品名称条目
msgctxt "STRINGS.NAMES.BEEFALO"
msgid "Beefalo"
msgstr "牛"

# 物品描述条目（噪声来源）
msgctxt "STRINGS.CHARACTERS.WILSON.DESCRIBE.BEEFALO"
msgid "It's a beefalo."
msgstr "这是一头牛。"
```

过滤策略：搜 `msgstr` 含 `pattern` 的条目块 → 只保留 `msgctxt` 含 `STRINGS.NAMES.` 的 → 输出 `BEEFALO → 牛`。含 `DESCRIBE` 的直接跳过。

**合并后的效果**：
- 工具数量：增加 3 个的计划 → 只增加 1 个
- DST 模式工具数：5 → 6（仍在可接受范围）
- 相比"不加合并"的 8 个工具，路由精度损失减少约 5-7 个百分点

### 数据源路径

三个数据源不在 `DST_SOURCE_DIR` 下，需要新增配置项 `DST_META_DIR` 或扩展 `InjectedToolArg`。待 `dst_data_lookup` 实现时确定。

## 历史决策（已废弃）

以下决策在 2026-07-04 的精简中因 `create_agent()` 和 Middleware 的移除而失去对照物。保留在此处以供考古。

### 决策 3：为什么同时提供三种 graph 构建方式？（废弃）

原内容见 git 历史 `9a992df`（阶段 B）至 `8838deb`（阶段 D）。三种构建方式为 `build_graph()`（手写 StateGraph + MemorySaver）、`build_graph_with_persistence()`（手写 + SqliteSaver）、`build_agent_with_middleware()`（`create_agent()` + 中间件）。

**废弃原因**：`create_agent()` 在 demo 入门阶段有临时教学价值，但其能力边界（框架固化的 agent loop、无法自定义节点）不足以支持远期控制台指令辅助功能的复杂编排。手写 StateGraph 的能力边界覆盖所有已知需求，三种方式并存的冗余不利于代码维护。

### 决策 4 & 5：LLM 节点命名差异（废弃）

原因为 `create_agent()` 编译后的 LLM 节点名为 `model`，手写版为 `agent`，测试中需分开检查。

**废弃原因**：`create_agent()` 已移除，不存在命名差异。

### 决策 9：中间件钩子的选择（废弃）

原因为使用 `before_model` / `after_model` / `wrap_tool_call` 三个 Middleware 钩子实现日志。

**废弃原因**：中间件是 `create_agent()` 专有机制，手写 StateGraph 无此体系。等价的日志逻辑已内嵌到 `src/agent/graph.py` 的 `agent_node` 闭包中（耗时 + token 用量 + tool_call 名称统计）。
